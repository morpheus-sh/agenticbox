# AgenticBox — Pricing & Growth Strategy

## Core Thesis

Agents are employees. Workplaces are infrastructure.

If you price it like a developer tool, you'll compete with sandboxes and open-source runtimes.
If you price it like identity/governance infrastructure, you can become part of the stack companies standardize on.

---

## Pricing

### Open Source Core (Current)

Keep:
- MIT/Apache-2.0
- Local runtime
- Docker sandboxing
- Basic permissions
- Agent packages

This maximizes adoption. The goal isn't revenue here.

The goal is: "Every serious agent developer knows what AgenticBox is."

### Pro ($20–50/month)

Individual builders.

Features:
- Desktop app
- Cloud sync
- Session history
- Advanced policies
- Browser profiles
- Hosted package registry

This is mostly for indie developers and small teams.

### Team ($15–30 per agent/month)

This is where things get interesting.

Not per user. **Per active agent identity.**

Example:
- 10 humans
- 50 agents
- Pay for 50 agent employees.

Features:
- Shared workplaces
- Team governance
- Approval workflows
- Audit retention
- Role templates

### Enterprise ($10k–100k+/year)

This is likely the real business.

Sell:
- Agent identity management
- Compliance
- Audit trails
- Credential provisioning
- Policy engines
- SSO integration

The buyer is not the AI engineer. The buyer is:
- Security
- IT
- Compliance
- Platform engineering

This is the same playbook that helped companies like Okta become infrastructure rather than developer tools.

---

## The Real Monetization Opportunity

Not sandboxes. **Identity.**

If AgenticBox becomes the place where:
- agents get credentials
- agents get browser identities
- agents get budgets
- agents get permissions
- agents get audit histories

then AgenticBox becomes the control plane.

That's much harder to replace than container execution.

---

## Growth Strategy

### Phase 1: Win Developers

Current strategy is good. Open source. GitHub-first. CLI-first.

Target:
- AI engineers
- agent builders
- OSS contributors

Goal: 10,000 developers know the concept of an "agent workplace."

Not revenue. Mindshare.

### Phase 2: Create the Category

Don't market sandboxing. Market: **Agent Workplace**.

Every blog post should hammer:
- Agents need identities
- Agents need permissions
- Agents need accountability
- Agents need workplaces

You want people saying: "We're deploying agents." followed immediately by: "What's their workplace?"

### Phase 3: Security-Led Expansion

This is where many AI startups miss the opportunity.

The buyer isn't the developer anymore. The buyer becomes:
- Security teams
- CISOs
- Compliance teams

Their nightmare: "An autonomous AI agent has production access."

Your pitch: "Every agent has a badge, permissions, and an audit trail."

That resonates far more than "multi-agent orchestration."

### Phase 4: Become the Default Runtime

The dream scenario:
- Agent frameworks become replaceable.
- Agent workplaces become mandatory.
- People choose different LLMs, different frameworks, different orchestration systems.
- But every agent runs inside AgenticBox.

That's the equivalent of becoming a layer in the stack.

---

## The Most Important Growth Loop

Create an ecosystem around agent packages.

```
agenticbox run recruiter
agenticbox run researcher
agenticbox run accountant
agenticbox run analyst
```

Every package:
- references AgenticBox
- requires AgenticBox
- spreads AgenticBox

That's essentially the same dynamic that made package managers, container registries, and developer platforms grow.

---

## What to Avoid

Avoid positioning as:
- "another agent framework"
- "another orchestration platform"
- "another AI operating system"

Those markets are crowded and confusing.

The strongest positioning remains: **AgenticBox gives AI agents a workplace.**

It's simple, memorable, and naturally expands into permissions, identity, governance, and compliance without changing the story.
